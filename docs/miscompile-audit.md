# rustc 1.97.0 aggregate-copy miscompile — audit & root cause

**Date:** 2026-07-15 · **Toolchain:** rustc 1.97.0 (2d8144b78 2026-07-07), bundled LLVM 22.1.6
**Status:** root cause isolated and reproduced; one *live* miscompiled site confirmed in the
shipped kernel binary; ranked fix list below. **No production code was changed** in this
checkpoint (`cargo check --workspace` and `cargo xtask qemu-test` re-verified green, 16/16).

---

## 1. The narrowed trigger — confirmed, not hypothesized

The three historical bugs (vmas slot store, `[u32; 32]` children snapshot, `Process`
placement-write) and the live corruption found during this audit are all one bug with one
precise trigger:

> **`-C target-feature=+sse…` combined with x86_64-unknown-none's default `+soft-float`
> breaks LLVM 22.1.6's lowering of *scalarized* aggregate copies.** The copy is emitted as
> single-byte loads/stores whose destination (and source) offsets step by **2** — the data
> is smeared across twice its extent, only even offsets written, odd bytes never stored
> (they keep whatever garbage was there). Any aggregate copy that instead lowers to a
> `memcpy` **call**, or is written as an element-wise loop, is unaffected.

Established by the `miscompile-repro/` matrix (isolated crate, kernel-identical profile:
`opt-level=3, lto="fat", codegen-units=1, panic="abort"`, kernel rustflags):

| Axis | Result |
|---|---|
| Host x86_64-pc-windows-msvc, same profile | clean, all 13 cases, all profiles |
| x86_64-unknown-none + kernel rustflags | **shredded** (cases a, b, e, f8, f16 — every `[u32; N≤32]` copy) |
| opt-level 0 / 1 / 2 / 3 | **all shredded** (o0 shreds *more* sites) — opt-level irrelevant |
| lto = fat / thin / off; cgu = 1 / 16 | **all shredded** — LTO/CGU irrelevant |
| flag bisection | `-C target-feature=+sse2` (or `+sse4.1`/`+sse4.2`) **alone** reproduces; `code-model=kernel`, `no-redzone`, `force-frame-pointers`, `relocation-model=static`, `+cmpxchg16b`, `+xsave` each alone: clean |
| `+sse4.2,-soft-float` (all other kernel flags kept) | **clean** |
| `[u32; 64]` (256 B) and every copy ≥ 512 B | lowered to a `memcpy` call → **correct** |

So the size dependence the project observed is real but secondary: it is LLVM's
scalarize-vs-memcpy decision (roughly ≤ 128–256 B scalarizes, context-dependent), and only
the scalarized path is broken. Pointer fields are **not** required (`[u32; 32]` shreds);
lock-guard derefs are **not** required (a plain local array copy, case e, shreds).

### Live corruption in the current kernel binary

A signature scanner (`miscompile-repro/src/bin/scan_asm.rs`) swept the **shipped kernel
ELF** (built 2026-07-14, the one that passes CI 16/16). It found exactly 8 stride-2 runs =
3 sites, all in the `Process` literal construction:

1. **`sys_fork` — `name: parent.name`** ([fork.rs:53](../crates/proc/src/fork.rs)): the
   `[u8; 16]` copy reads/writes only the 8 even bytes (parent→stack at
   `ffffffff80015972`, stack→child at `ffffffff80015af8`); odd bytes are never written
   anywhere in the function (verified exhaustively, all store widths). **Every forked
   child's name has 8 garbage bytes** — silent only because `Process.name` is currently
   write-only (nothing prints it).
2. **`sys_clone` (inlined into `proc_syscall_handler`) — `name: parent.name`**
   ([clone.rs:130](../crates/proc/src/clone.rs)): identical shredding at
   `ffffffff800115ab`/`ffffffff80011737`.
3. **`Process::create` (inlined into `kernel_main`)**: four stride-2 runs at
   `ffffffff8000764f`+ writing offsets 0xd54–0xd92 — SpinLock/struct **tail padding** and
   a few bytes past `size_of::<Process>()` but inside the owned 4 KiB frame. Harmless
   today by pure luck; same broken lowering.

Evidence excerpts: [`miscompile-evidence/kernel-sys_fork-name-shred.s`](miscompile-evidence/kernel-sys_fork-name-shred.s),
[`miscompile-evidence/repro-case_a-stride2-shred.s`](miscompile-evidence/repro-case_a-stride2-shred.s).
Full dumps are regenerated by `miscompile-repro/run-matrix.sh`.

**Honesty note:** the isolated repro did *not* corrupt at runtime on the host (different
target, different lowering), and the `Some(struct)` slot-store cases (c, d) and big-array
cases (h, i, j) compile correctly in isolation at opt≥1 — the historical bug #1/#3 sites
presumably shredded because of their exact surrounding code under fat LTO. The
target-side **assembly** is the reliable oracle, not host runtime checks. The scanner
detects only the stride-2 byte-store manifestation (the only one observed); a different
manifestation of the same LLVM bug would evade it.

## 2. Known rustc issue?

Yes — same family, same version. [rust-lang/rust#159035](https://github.com/rust-lang/rust/issues/159035)
("segfault on Rust 1.97.0", **I-miscompile, P-critical, A-LLVM, llvm-fixed-upstream**) is
filed against this exact rustc build and root-caused to LLVM 22.1.6's aggregate lowering
for types with padding. The LLVM fix is [llvm/llvm-project#208683](https://github.com/llvm/llvm-project/pull/208683),
backported to Rust in [rust-lang/rust#159106](https://github.com/rust-lang/rust/pull/159106)
for a **1.97.1 point release** (not yet available as of 2026-07-15).

Match caveat: #159035's reproducer (enum + `Option::map`, segfault on
x86_64-unknown-linux-gnu) manifests differently from our stride-2 shredding, and does not
mention soft-float. It is plausibly the same underlying padding/poison-model bug but **not
confirmed identical** — when 1.97.1 lands, re-run `miscompile-repro/run-matrix.sh` before
declaring victory. Our soft-float+SSE trigger stands on its own regardless.

## 4. Recommended defensive pattern

**Primary (fixes the class, not the symptom): change the build flags.** In
`.cargo/config.toml`, either append `-soft-float` to the feature list
(`+cmpxchg16b,+sse4.2,+xsave,-soft-float`) or drop `+sse4.2`. Both eliminate every
shredded copy in the repro under the full kernel flag set. Caveats, which is why this is
Phase 2 and not done here: `-soft-float` licenses LLVM to use SSE registers in kernel code
paths, so interrupt/syscall entry must be audited for FPU-state clobber (the kernel
already opted into `+sse4.2`, so this risk was partially accepted already); it changes the
float ABI of every crate. Alternatively/additionally: **move to 1.97.1 when it releases**
and re-run the matrix — but do not assume it fixes this manifestation (see §2 caveat).

**Secondary (code-level convention until the flag/toolchain fix is verified):** adopt one
helper in `kernel-sync` (or a tiny new `kernel-util`):

```rust
/// Copy `src` into `dst` element-by-element. Never lowered to a single
/// memcpy/scalarized aggregate copy — immune to the rustc 1.97.0
/// soft-float+SSE aggregate-copy miscompile (docs/miscompile-audit.md).
#[inline(never)]
pub fn copy_elems<T: Copy, const N: usize>(dst: &mut [T; N], src: &[T; N]) {
    for i in 0..N {
        dst[i] = src[i]; // per-element; each store is a plain typed store
    }
}
```

plus the review rule: **no whole-value reads/writes of arrays or array-bearing structs
through any pointer/guard** (`let x = *guard`, `a = b.field_array`, `arr[i] = Some(big)`)
— use `copy_elems`, element loops, or the existing repr(C) per-field offset writes.
`scan_asm` should be run against the kernel ELF in CI (it's fast) so any regression to
stride-2 lowering fails loudly instead of silently.

Per-element copies of `Copy` scalars are exactly what the three historical fixes already
do; this just names the pattern. (Caveat honestly stated: at `opt-level=0` the repro
shredded even more sites, and nothing formally guarantees LLVM never re-aggregates a
loop — the flag fix in #1 is the real fix; the helper is defense-in-depth plus greppable
intent.)

## 5. Fix-these-next (Phase 2 hand-off, in order — one verified change each)

1. **Flags: DONE (2026-07-15, checkpoint "soft-float + pipe/dup").** `.cargo/config.toml`
   now sets `target-feature=+cmpxchg16b,+xsave,-sse,-sse2,-sse3,-ssse3,-sse4.1,-sse4.2,
   -avx,-avx2`. Bisection confirmed the trigger is `+sse*` (any level, even `+sse2`)
   combined with x86_64-unknown-none's default `+soft-float`; dropping every SSE level is
   ABI-safe because this target's baseline is soft-float with only `fxsr` enabled. After
   the change: `scan_asm` on the rebuilt kernel reports **0 runs** (was 8), the kernel
   `.text` contains **zero `%xmm`/SSE instructions**, all 251 host tests pass, and
   `cargo xtask qemu-test` is green (now **17** lines — added a `fork: child name intact`
   check via a new `prctl(PR_GET_NAME)` syscall, the first-ever reader of `Process.name`;
   proven to have teeth: it prints `... SHREDDED ... - FAIL` when built with the old
   `+sse4.2` flags). Kept the three historical element-wise fixes as-is.
   - **FPU audit (Step 4):** the kernel is now soft-float-only, so it emits no SSE and
     needs no FPU save/restore for its own code. `xsave64`/`xrstor64` exist in
     `arch-x86_64/src/xsave.rs` but have **no callers** (gc-sectioned out); context switch,
     IDT stubs, and the syscall trampoline never touch FPU state. **Pre-existing gap
     (NOT fixed here, out of scope):** userspace SSE/FPU state is not saved across context
     switches, so once a real userspace libc uses SSE it can be clobbered by an unrelated
     thread switch. No current userspace program uses SSE (the init2 demo is pure integer),
     so this is latent. Wiring `xsave64`/`xrstor64` into `switch_context` (with an
     XSAVE-area allocation per thread) is future work.
   - **Harness fix (incidental):** `cargo xtask qemu-test` was crashing QEMU host-side
     (`0xc0000005`) — the vvfat `rw` ESP mount let OVMF persist NvVars through QEMU's
     unmaintained vvfat write path and corrupt the QEMU process on Windows. Fixed by
     mounting the ESP with `snapshot=on` (guest writes land in a throwaway overlay). This
     was a pre-existing fragility unrelated to the flag change; it only surfaced now
     because the repeated rebuild/boot cycles accumulated NvVars writes.
2. **fork.rs:53 / clone.rs:130 `name` copies** (live corruption): element-wise copy after
   the placement write; verify by re-scanning the ELF (the 8 stride-2 runs must be gone)
   and a QEMU test that forks and echoes the child's name.
3. **address_space.rs:432 `child.vmas = self.vmas`**: rewrite element-wise (bug #1's array
   was never comprehensively fixed) + add a QEMU test: file-backed `mmap` → `fork` →
   child reads the mapping.
4. **fd.rs `clone_for_fork`**: element-wise + a QEMU test forking with ≥4 open fds.
5. **sig_table guard copies** (signal.rs ×2, fork.rs, clone.rs, exec.rs): borrow instead
   of copy where possible, else element-wise.
6. **signal.rs sig_frame/`write_signal_frame` 144 B copies**: element-wise or field-wise.
7. Sweep rows 10–14 with `copy_elems`/field-wise stores opportunistically, lowest priority
   — each is scan-clean today; re-scan after **every** step above.
